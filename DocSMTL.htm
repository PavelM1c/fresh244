Библиотека технологии сервиса
Версия 1.1.1

Состав конфигурации

Инструментария конфигурации БТС недостаточно для работы прикладных решений в модели сервиса. На данном этапе в обязательном порядке требуется внедрение части подсистем инструментария БСП. При внедрении БТС необходимо использовать версию 8.3.10 и выше платформы «1С:Предприятие».

Все объекты библиотеки поставляются в составе ряда подсистем включенных в корневую подсистему ТехнологияСервиса

Базовая функциональность

Подсистема Базовая функциональность включает в себя набор вспомогательных механизмов, которые используются другими подсистемами библиотеки. Необходимо внедрять данную подсистему в состав конфигурации при внедрении любой другой подсистемы БТС.

Выгрузка / загрузка данных

Подсистема Выгрузка / загрузка данных является вспомогательной для подсистемы Выгрузка / загрузка областей данных и не предполагает самостоятельного использования для конфигураций, которые должны использоваться в модели сервиса. Тем не менее, она может отдельно внедряться в конфигурации, не содержащие механизмов развертывания в модели сервиса, для обеспечения возможности выгрузки данных для перехода в сервис (например, в предыдущие редакции конфигураций).

Информационный центр

Подсистема Информационный центр предоставляет возможность пользователю прикладного решения получить методическую поддержку непосредственно из прикладного решения.

Информационный центр предоставляет возможности:

  1. Отображение ссылок на полезные ресурсы и статьи.
  2. Отправка писем в техническую поддержку и просмотр переписки по ним.
  3. Поиск информации на сайте ИТС.
  4. Отображение недоступности ресурсов сервиса.
  5. Просмотр и добавление идей (пожеланий), голосование за предложенную идею, комментирование пожеланий.
  6. Отображение информационных ссылок на любые внешние ресурсы в формах прикладного решения.

Работа в модели сервиса

Базовая функциональность

Подсистема Базовая функциональность в модели сервиса включает в себя набор вспомогательных механизмов, которые используются другими подсистемами группы подсистем Работа в модели сервиса. Необходимо внедрять данную подсистему в состав конфигурации при внедрении любой другой подсистемы этой группы.

Выгрузка / загрузка областей данных

Подсистема Выгрузка / загрузка областей данных предоставляет возможность выполнять выгрузку данных приложения в xml-файлы, которые в дальнейшем могут быть загружены в другую информационную базу или область данных. В частности, с использованием механизмов подсистемы реализуются:
  1. Перенос данных пользователя из локальной версии конфигурации в приложение в сервисе;
  2. Перенос данных пользователя из сервиса в локальную версию конфигурации;
  3. Создание резервных копий приложений в сервисе (и восстановление приложений из резервных копий).
Подсистема Выгрузка / загрузка областей данных является обязательной для внедрения в конфигурации, которые должны поддерживать возможность развертывания в модели сервиса.
При внедрении подсистемы обязательным является внедрение подсистемы Выгрузка / загрузка данных.

Настройка интерфейса OData

Подсистема Настройка стандартного интерфейса OData позволяет реализовать рекомендуемую схему использования для приложений сервиса автоматического REST-сервиса, реализованного в версии 8.3.5 технологической платформы 1С:Предприятие (см. http://its.1c.ru/db/v83doc#bookmark:dev:ti000001358).
В составе подсистемы поставляются:
  • Роль УдаленныйДоступСтандартныйИнтерфейсOData, предназначенная для авторизации сторонних программ при доступе к автоматическому REST-сервису
  • К составу роли предъявляются следующие требования:
    • Роль должна содержать права на чтение и запись объектов, разделенных разделителем ОбластьДанныхОсновныеДанные
    • Роль должна содержать права на чтение объектов, не разделенных разделителем ОбластьДанныхОсновныеДанные, но ссылки на которые используются в объектах, разделенных разделителем ОбластьДанныхОсновныеДанные

    • Ошибки в настройке состава роли диагностируются как ошибки при разработке конфигурации и прерывают выполнение неразделенной фазы обновления информационной базы
  • Состав роли может поддерживаться разработчиком как вручную, так и заполняться автоматически с помощью внешней обработки ИнструментыРазработчикаЗаполнениеРолиДляСтандартногоИнетрфейсаOData.epf, поставляемой в составе библиотеки

  • Обработка НастройкаСтандартногоИнтерфейсаOData, позволяющая администратору области данных (пользователь с ролью ПолныеПрава)

    Подсистемы ЦКК в модели сервиса

    Подсистема включает набор механизмов для использования возможностей Центра Контроля Качества (ЦКК) для централизованного мониторинга прикладных инцидентов и показателей.
    Подсистема предоставляет такие возможности:
    1. Хранение данных для доступа к web-сервисам ЦКК.
    2. Определение списка типов инцидентов, которые прикладное решение будет передавать в ЦКК и поведения типов. Можно ограничивать максимальную частоту отсылки инцидентов каждого типа и определять процедуры для автоматического закрытия неактуальных инцидентов.
    3. Отсылка инцидентов ранее определенных типов в ЦКК. Поддерживается синхронная и асинхронная отсылка.
    4. Отсылка в ЦКК массива прикладных показателей. В ЦКК такие показатели можно выводить на панели мониторинга и отслеживать их пороговые значения, уведомляя ответственных по SMS и EMail.
    Подсистема поддерживает создание инцидентов в прикладном коде, в котором выполняется действие, могущее вызвать инцидент. Также поддерживается список процедур, которые будут вызываться периодически и проверять состояние информационной базы на наличие ситуаций, требующих мониторинга.
    Компоненты подсистемы не работают в файловом режиме работы информационной базы, в этом случае вызовы методов будут проигнорированы. Также вызовы будут проигнорированы, если не установлены данные для доступа к web-сервисам ЦКК. Эти данные устанавливаются с помощью события SetIBParams, которое прикладным приложениями в сервисе централизованно рассылает менеджер сервиса.

    Управление тарифами

    Подсистема Управление тарифами в модели сервиса позволяет тарифицировать услуги прикладной конфигурации, которые входят в состав самой конфигурации. Прикладному разработчику в составе данной библиотеки предоставлено:
    • Протокол регистрация услуг конфигурации, которые можно будет тарифицировать Провайдеру сервиса
    • API, с помощью которого прикладной разработчик может взаимодействовать с системой тарификации, расположенной в Менеджере сервиса.
    В системе тарификации есть два основных понятия:
    • Услуга сервиса - услуга, которая может тарифицироваться в сервисе.
    • Поставщик услуг сервиса - поставщик (провайдер или владелец) услуг.
    Услуги бывают трех видов:
    • Безлимитная - услуга сервиса, которая не имеет количественного признака и предоставляет доступ к какому-то функционалу непосредственно.
    • Уникальная - услуга сервиса, которая имеет количественный признак и предоставляет доступ к какому-то функционалу при наличии свободной лицензии данной услуги.
      * Лимитированная - услуга сервиса, которая имеет количественный признак и предоставляет доступ к какому-то функционалу при наличии свободной лицензии данной услуги.
    Отличие уникальной услуги от лимитированной:
    • Уникальная услуга хранит отдельную запись для каждой занятой лицензии.
    • Лимитированная услуга хранит одну запись для нескольких занятых лицензий. В этой записи хранится количество занятых лицензий.
    Для тарификации произвольного функционала в конфигурации, прикладной разработчик должен зарегистрировать свою услугу по определенному протоколу: в общем модуле "ТарификацияПереопределяемый" данной библиотеки добавить запись вида: 
    МассивУслуг = Новый Массив;

    Услуга = Новый Структура;
    Услуга.Вставить("Идентификатор", "ПерваяТестоваяУслуга"); // Идентификатор - это Строка(50).
    Услуга.Вставить("Наименование", "Получить тестовую услугу #1"); // Наименование - это Строка(150).
    Услуга.Вставить("ТипУслуги", Перечисления.ТипыУслуг.Безлимитная);
    МассивУслуг.Добавить(Услуга);

    Услуга = Новый Структура;
    Услуга.Вставить("Идентификатор", "ВтораяТестоваяУслуга");
    Услуга.Вставить("Наименование", "Получить тестовую услугу #2");
    Услуга.Вставить("ТипУслуги", Перечисления.ТипыУслуг.Лимитированная);
    МассивУслуг.Добавить(Услуга);

    НовыйПоставщик = Новый Структура;
    НовыйПоставщик.Вставить("Идентификатор", "ПоставщикТестовыхУслуг"); // Идентификатор - это Строка(50).
    НовыйПоставщик.Вставить("Наименование", "Поставщик тестовых услуг"); // Наименование - это Строка(150).
    НовыйПоставщик.Вставить("Услуги", МассивУслуг);

    ПоставщикиУслуг.Добавить(НовыйПоставщик);
    В общем модуле "Тарификация" прикладному разработчику предоставлен программный интерфейс, с помощью которого он может взаимодействовать с системой тарификации Технологии Fresh. Ниже предоставлен список процедур и функций данного API:
    • ЗарегистрированаЛицензияБезлимитнойУслуги - проверяет, позволяет ли система тарификации сервиса использование указанной безлимитной услуги.
    • ЗарегистрированаЛицензияУникальнойУслуги - проверяет, зарегистрирован ли в системе тарификации сервиса указанный идентификатор лицензии на использование указанной уникальной лимитированной услуги.
    • ЗанятьЛицензиюУникальнойУслуги - выполняет попытку получить лицензию на использование уникальной услуги в системе тарификации сервиса.
    • ОсвободитьЛицензиюУникальнойУслуги - выполняет попытку освободить лицензию на уникальную услугу в системе тарификации сервиса.
    • ЗанятьЛицензииЛимитированнойУслуги - выполняет попытку получить лицензии на использование лимитированной услуги в сервисе.
    • ОсвободитьЛицензииЛимитированнойУслуги - выполняет попытку освободить лицензии на использование лимитированной услуги в сервисе.
    • ПодтвердитьОперацию - выполняет подтверждение ранее запрошенной операции с лицензиями (получение или освобождение).
    • ОтменитьОперацию - выполняет отмену ранее запрошенной операции с лицензиями (получение или освобождение).

    Внешний программный интерфейс

    Подсистема Внешний программный интерфейс предназначена для работы с API внешних по отношению к прикладной информационной базе компонентов сервиса 1cFresh, в частности, с внешним API Менеджера сервиса. Программный интерфейс, предоставляемый подсистемой прикладному разработчику, представляет собой "обертку" (wrapper) для методов внешнего программного интерфейса и позволяет вызывать эти методы с минимальными усилиями, без обращения к инфраструктурному слою (прокси веб-сервисов, сериализация и проверка параметров и так далее).
    В текущей версии БТС реализованы такие "обертки" только для некоторых методов внешнего API Менеджера сервиса, в следующих версиях планируется существенно расширить их состав.
    Подробнее о внешнем API Менеджера сервиса можно прочитать в документе "1С:Технология разработки 1cFresh" из комплекта технической документации, поставляемого с Технологией.

    Интеграция объектов областей данных

    Подсистема содержит объекты для организации транспорта данных при интеграции приложения с внешними учетными системами.
    Обеспечивает транспорт и не зависит от формата передаваемых данных, не знает об объектах прикладной логики, оперирует только идентификаторами объектов. Таким образом, возможна передача не объектных данных, например отчетов.
    Подсистема для своей работы использует другие подсистемы:
    • Механизм работы с логическими хранилищами подсистемы Передача данных
    • Подсистему Очередь заданий внешний интерфейс
    • Подсистему Файлы областей данных.

    Программный интерфейс

    Программный интерфейс содержится в общем модуле ИнтеграцияОбъектовОбластейДанных. Объединяет интерфейс логического хранилища и интерфейс для работы с объектами подсистемы.
    Определяемый тип УчетныеСистемыИнтеграцииОбластейДанных предназначен для определения метаданных учетных систем.
    По ссылке на объект этого типа будут сохраняться настройки в безопасное хранилище:
    • Пароль (Строка) - пароль аутентификации в сервисе приема оповещений (используется при basic-аутентификации)
    • ПарольСертификата (Строка) - пароль сертификата (используется, если задано свойство ИспользоватьСертификат);
    • ДанныеСертификата (ДвоичныеДанные) - двоичные данные сертификата в base64 (используется, если задано свойство ИспользоватьСертификат)
    • КлючПодписи (Строка) - секретное слово, для подписи отправляемых данных. Подпись выполняется с помощью алгоритма HMACSHA256.
    Также этот тип определяет:
    • разрез хранения объектов к отправке в регистре сведений ОбъектыКОтправке
    • хранение настроек учетных систем в регистре сведений НастройкиУчетныхСистем.
    В регистре сведений НастройкиУчетныхСистем хранятся настройки для определения учетной системы и для отправки оповещений.
    • Пользователь (СправочникСсылка.Пользователи) - для определения учетной системы по авторизованному пользователю при обращении к API.
    • ОповещатьОбИзменениях (Булево) - признак использования оповещений при создании/изменении данных в приложении
    • АдресСервиса (Строка) - адрес сервиса приема оповещений об изменениях
    • СпособАутентификации (ПеречислениеСсылка.СпособыАутентификации) - способ аутентификации в сервисе приема оповещений
    • Логин (Строка) - логин аутентификации в сервисе приема оповещений (используется при basic-аутентификации)
    • ИспользоватьСертификат (Булево) - признак использования сертификата при установке соединения с сервисом приема оповещений
    • ИмяСертификата (Строка) - имя файла сертификата
    • ПодписыватьДанные (Булево) - признак использования подписи данных при отправке их в сервис приема оповещений

    Интерфейс логического хранилища

    1. Описание - метод не поддерживается

    4. confirm - для подтверждения получения данных от приложения

    Подсистема Автоматизация процессов предоставляет простой программный интерфейс, позволяющий выполнить следующие операции:

    • Выполнить в области команду дополнительной обработки.
    • Инициировать передачу файла в другую область.
    • Инициировать передачу квитанции о получении файла в область-отправитель.
    Каждая операция представлена отдельным методом программного интерфейса. Также подсистема содержит переопределяемую часть, которая позволяет прикладному разработчику описать собственные обработчики следующих событий:

    • Получение квитанции об ошибке передачи файла в другую область.

    Миграция приложений

    Подсистема Миграция данных предоставляет возможность выполнять миграцию (прозрачный перенос) данных напрямую между источником и приемником, без использования промежуточных файлов. Перенос данных в этом случае не требует монопольного доступа и может выполняться во время работы пользователей. В частности, с использованием механизмов подсистемы реализуются: Перенос данных пользователя из локальной версии конфигурации в приложение в сервисе; Перенос приложения из одного сервиса в другой.
    Подсистема Миграция данных является обязательной для внедрения в конфигурации, которые должны поддерживать возможность перенос данных без установки монопольного режима.

    Порядок внедрения:

    1. Состав плана обмена "Миграция приложений" заполнить всеми объектами.
    2. Вынести команду "Обработка.МиграцияПриложения.Команда.ПерейтиВСервис" в интерфейс в раздел Администрирование или другой подходящий.
    3. При необходимости использовать обработчики ПриОпределенииСервисов, ПриВыгрузкеОбъекта, ПриЗагрузкеОбъекта в общем модуле МиграцияПриложенийПереопределяемый.

    • Можно подключать различные обработчики данных (хэндлеры) и расширять существующие.

    В этом случае система "1С:Предприятие" создает новый сеанс, выполняет аутентификацию, устанавливает разделители, выполняются все необходимые обработчики событий. Если система не может создать новый сеанс, то клиент получит ошибку 406 Not Acceptable.     При необходимости использовать ранее созданный сеанс, необходимо в HTTP-запросе к системе "1С:Предприятие" указать идентификатор ранее созданного сеанса: Cookie: ibsession=<ID сеанса>. Если в запросе указывается идентификатор сеанса, который ранее не создавался или был завершен, клиент получает ошибку 400 Bad Request.

    Если в HTTP-запрос не содержит заголовка IBSession, то сеанс создается и завершается при каждом вызове интернет-сервиса.

    Если в процессе использования сеанса в HTTP-запросе изменяются значения разделителей или безопасный режим сеанса, то новые параметры игнорируются и сеанс будет использовать те значения, которые были указаны при старте сеанса.
    IBSession finish В этом случае система "1С:Предприятие" завершает сеанс, который указан в запросе, одновременно с командой завершения сеанса: Cookie: ibsession=<ID сеанса>. Завершение сеанса произойдет автоматически, если в этом сеансе не выполнялось никаких действий за время жизни сеанса.

    Проверка и корректировка данных

    Подсистема Проверка и корректировка данных предоставляет прикладным разработчикам возможность встраивания в конфигурацию обработок, выполняющих произвольную проверку и (при необходимости) модификацию данных области. Подсистема представляет собой "каркас" и реализует следующие возможности:

    • Предоставить пользователю возможность запустить выбранные проверочные обработки
    • Запускать проверочную обработку в одном из двух режимов: "Только проверка", "Проверка и корректировка"
    • Вызывать форму настройки проверочной обработки
    • Формирование протокола проверки/корректировки, сохранение последнего сформированного протокола в информационной базе

    Эти возможности реализованы в одноименной обработке Проверка и корректировка данных. При встраивании подсистемы необходимо вывести эту обработку в командный интерфейс администратора области и/или информационной базы.
    В качестве примера проверочной обработки можно использовать обработку Проверка измерений в регистрах бухгалтерии, которая входит в поставку БТС. В проверочной обработке должны быть реализованы три экспортных метода модуля менеджера.

    Настройка состава объектов, исключаемых из выгрузки / загрузки данных

    Если в конфигурации существуют объекты метаданных, которые используются только в локальном режиме работы, или наоборот (только в сервисе), необходимо указать данные объекты метаданных в процедуре ВыгрузкаЗагрузкаДанныхПереопределяемый.ПриЗаполненииТиповИсключаемыхИзВыгрузкиЗагрузки. В данную процедуру передается массив, в который внутри процедуры требуется добавить значения типа ОбъектМетаданных, соответствующие таким объектам метаданных

    Настройка состава объектов, для которых при загрузке данных выполняется сопоставление ссылок с уже существующими в ИБ-приемнике данных

    При использовании в конфигурации разделенных объектов метаданных, в свойствах которых хранятся ссылки на неразделенные ссылочные объекты метаданных, требуется обеспечить возможность сопоставить ссылки на неразделенные объекты метаданных в ИБ, из которой была выполнена выгрузка данных, с аналогичными ссылками в ИБ, в которую загружаются данные.
    Данное требование связано с тем, что при загрузке данных в сервис будет выполняться загрузка только разделенных данных, а все ссылки на неразделенные данные станут "битыми".
    Для того, чтобы обеспечить данное требование, необходимо для каждого неразделенного ссылочного объекта метаданных, ссылки на элементы которого могут использоваться в разделенных данных, выполнить следующие действия:
    1. Указать данный объект метаданных в процедуре ВыгрузкаЗагрузкаДанныхПереопределяемый. ПриЗаполненииТиповОбщихДанныхПоддерживающихСопоставлениеСсылокПриЗагрузке(). В данную процедуру передается массив, в который внутри процедуры требуется добавить значение типа ОбъектМетаданных, соответствующее неразделенному объекту метаданных
    2. В модуле менеджера объекта метаданных расположить экспортируемую функцию ПоляЕстественногоКлюча(), которая будет возвращать массив, содержащий имена полей объекта, сочетание которых образует естественный ключ для этого объекта метаданных. При выгрузке данных сочетание полей естественного ключа и ссылки элемента в исходной ИБ будет отдельно сохранено в выгрузке данных, а при загрузке - прочитано и сопоставлено со значениями полей естественного ключа в ИБ, в которую выполняется загрузка данных. Также, при выгрузке данных будет выполняться уникальность сочетаний значений полей естественного ключа в выгружаемых данных. В качестве полей, входящих в состав естественного ключа, допускается указывать имена реквизитов, стандартных реквизитов и общих реквизитов объекта метаданных. Пример функции ПоляЕстественногоКлюча() для справочника, элементы которого могут быть сопоставлены по значению стандартного реквизита Код:
      // Возвращает реквизиты справочника, которые образуют естественный ключ для элементов справочника.
      // Возвращаемое значение: 
      //  Массив(Строка) - массив имен реквизитов, образующих естественный ключ.
      //
      Функция ПоляЕстественногоКлюча() Экспорт
          Результат = Новый Массив();
          Результат.Добавить("Код");
          Возврат Результат;
      КонецФункции;
    В составе подсистемы поставляются механизмы контроля выполнения этого требования по настройке подсистемы:
    1. При любом изменении номера версии конфигурации будет выполняться неразделенный (ОбщиеДанные = Истина) обработчик обновления, который будет проверяться структуру метаданных на наличие в свойствах разделенных объектов метаданных ссылок на элементы неразделенных ссылочных объектов метаданных, которые не указаны в процедуре ВыгрузкаЗагрузкаДанныхПереопределяемый.ПриЗаполненииТиповОбщихДанныхПоддерживающихСопоставлениеСсылокПриЗагрузке(). В случае наличия таких ссылок - будет генерироваться исключение с указанием объекта метаданных, на который обнаружена ссылка, объекта метаданных, в котором обнаружена ссылка, и имени свойства, в котором обнаружена ссылка. При этом использование для свойств разделенных данных типов вида ЛюбаяСсылка, СправочникСсылка., ДокументСсылка. и т.д. не будет диагностироваться как ошибка, т.к. на практике в качестве значения таких свойств могут использоваться только ссылки на разделенные объекты метаданных;
    2. Проверка значений таких свойств будет выполняться при выгрузке данных, и при обнаружении в выгружаемых разделенных данных ссылок на элементы неразделенных ссылочных объектов метаданных, которые не указаны в процедуре ВыгрузкаЗагрузкаДанныхПереопределяемый.ПриЗаполненииТиповОбщихДанныхПоддерживающихСопоставлениеСсылокПриЗагрузке() - будет генерироваться исключение с указанием объекта метаданных, на который обнаружена ссылка, объекта метаданных, в котором обнаружена ссылка и имени свойства, в котором обнаружена ссылка.
    В экзотических случаях, в которых не требуется обеспечивать сопоставление ссылок на неразделенные ссылочные объекты метаданных (например, когда в разделенных данных используются ссылки на предопределенные элементы неразделенных ссылочных объектов метаданных, либо пустые ссылки), нужно указать объект метаданных в процедуре ВыгрузкаЗагрузкаДанныхПереопределяемый.ПриЗаполненииТиповОбщихДанныхНеТребующихСопоставлениеСсылокПриЗагрузке(). В данную процедуру передается массив, в который внутри процедуры требуется добавить значение типа ОбъектМетаданных, соответствующее неразделенному объекту метаданных. В этом случае данный объект метаданных будет исключен из состава вышеперечисленных проверок.

    Особенности при внедрении в конфигурации, не содержащие механизмов развертывания конфигурации в модели сервиса

    Если подсистема внедряется в конфигурацию, которая не должна содержать механизмов использования конфигурации в модели сервиса (и в частности, не должна содержать подсистемы Выгрузка / загрузка областей данных) - предполагается, что внедрение выполняется в предыдущую версию конфигурации, из которой необходимо обеспечить возможность выгрузки данных для перехода в сервис (важно, что в этом случае исходная конфигурация должна поддерживать возможность обновления до версии, используемой в сервисе, с помощью 1cv8.cf или 1cv8.cfu файлов).
    В этом случае, помимо вышеприведенных шагов по настройке подсистемы, требуется разместить в пользовательском интерфейсе команду выгрузки данных для переноса в сервис, при выполнении которой будет выполняться код вида: 
    // Выгружает данные из локальной информационной базы для перехода в сервис.

        ПараметрыВыгрузки = Новый Структура();

        ПараметрыВыгрузки.Вставить("ВыгружатьПользователей", Истина);
        ПараметрыВыгрузки.Вставить("ВыгружатьНастройкиПользователей", Истина);

    	Возврат ИмяФайла;

    Также, рекомендуется указывать в процедуре ВыгрузкаЗагрузкаДанныхПереопределяемый.ПриЗаполненииТиповОбщихДанныхПоддерживающихСопоставлениеСсылокПриЗагрузке() объекты метаданных, которые после обновления до версии, используемой в сервисе, окажутся неразделенными ссылочными объектами метаданных, ссылки на элементы которых могут использоваться в разделенных данных, и реализовывать для них в модуле менеджера функцию ПоляЕстественногоКлюча(). Это позволит выполнять контроль уникальности сочетаний значений полей естественного ключа при выгрузке данных и, в случае нарушения уникальности - уведомлять пользователя о том, что перед выгрузкой данных необходимо устранить неуникальные значения. В противном случае, данные пользователя будут выгружены в zip, но могут быть после этого некорректно загружены в информационную базу в сервисе.
